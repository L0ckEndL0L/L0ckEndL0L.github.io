Building the Foundation for Year-Over-Year Revenue Insights

A breakdown of how I improved my Revenue Management System by adding year-over-year comparison logic, variance calculations, validation, and missing-data handling.

View Repository Back to Posts

Author: Thomas Hayden

Date: May 30, 2026

Project: Flexible Revenue Management System for Hotel Pricing Decision Support

Monthly Goal

The goal for this month was to move the Revenue Management System from basic data handling into a more useful decision-support tool. The main issue I focused on was that hotel performance data is not very helpful if it only shows current numbers. Revenue managers need to understand how current performance compares against prior-year results so they can recognize trends, identify risk, and make better pricing decisions.

My strategy was to strengthen the project’s data pipeline, improve validation, and build year-over-year comparison logic that could calculate meaningful hotel performance metrics even when the data was incomplete or inconsistent.

Features Developed This Month

This month, I focused on building and improving the year-over-year comparison features in the Revenue Management System. I updated the project so current hotel performance rows can be matched against prior-year data using stay dates. The system first attempts to match the exact prior-year date, and when that is not available, it can fall back to a calendar-based month-and-day comparison. This makes the tool more flexible when working with real hotel data that may not always export perfectly.

I also added variance calculations for core hotel metrics, including occupancy, ADR, RevPAR, rooms sold, and room revenue. These calculations allow the system to show the difference between current performance and prior-year performance using both absolute variance and percentage variance.

Another major improvement was handling missing prior-year data. Instead of breaking the calculations or creating misleading numbers, the system now flags rows where prior-year data is incomplete or unavailable. This makes the reporting more trustworthy because it clearly separates valid comparisons from areas where more historical data is needed.

The main tools used this month included Python, pandas, pytest, and the existing project files such as yoy.py, main.py, app.py, validate.py, test_yoy.py, and test_validate.py.

What Went Right

The biggest success this month was getting the year-over-year comparison logic working in a way that supports real hotel decision-making. The project is now more than a basic calculator because it can compare current performance against historical performance and identify whether a property is pacing ahead or behind.

Another positive outcome was improving the validation process. Adding required-field checks helped make the pipeline more reliable and reduced the chance of bad data creating incorrect results. I also created hand-check tests to verify that the formulas were working correctly.

What Went Wrong

The biggest challenge was working around data limitations. Some of the available property data did not fully support the time ranges needed for stronger year-over-year comparisons, so I had to request updated data from other properties.

Another challenge was making sure the logic did not become too rigid. Hotel data can be messy, and a comparison system that only works with perfect data would not be useful in a real-world environment.

How I Can Improve Moving Forward

Moving forward, I can improve by continuing to test with larger and more varied datasets. The more property data I can run through the system, the better I can validate the results and identify edge cases.

I also want to continue improving how the results are displayed in the app so the user can quickly understand the comparison without needing to inspect raw output files.

What I Learned This Month

This month helped connect several previous course topics directly to my capstone project. Python programming, data cleaning, validation, testing, and software design all came together in one system.

I also learned that building a practical software tool is not just about making formulas work. The system also has to handle real-world problems, such as missing data, inconsistent exports, and unclear historical records.

Readiness for the Next Phase

I am ready to move into the next phase of development. The core year-over-year logic, variance calculations, validation improvements, and hand-check testing are now in place. The next step is to continue refining the system with updated property data, improve the dashboard outputs, and make sure the final project clearly demonstrates how the Revenue Management System supports better hotel pricing decisions.
