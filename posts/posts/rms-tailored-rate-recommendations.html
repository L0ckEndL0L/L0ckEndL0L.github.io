Preparing RateAnchor for Tailored Hotel Revenue Recommendations

A breakdown of how I moved my Revenue Management System closer to a deployable hotel pricing decision-support app by adding tailored recommendation logic, improving demo data, and preparing the Streamlit interface for review.

View Repository Back to Posts

Author: Thomas Hayden

Date: June 23, 2026

Project: RateAnchor Revenue Management System

Monthly Goal

The goal for this month was to make the Revenue Management System more useful as a real hotel pricing decision-support tool. Earlier versions focused on baseline forecasting, KPI calculations, validation, and year-over-year comparisons. This month, I focused on making the recommendations more flexible, more explainable, and easier to demonstrate through the web app.

I also wanted the project to be ready for capstone review and deployment. That meant improving the Streamlit interface, making sure the demo data actually worked, and preparing the project so someone reviewing it could load sample data, run a simulation, and understand the pricing logic without needing to manually configure every file.

Features Developed This Month

The largest feature added this month was the tailored recommendation layer. I built new logic that sits on top of the baseline pricing output and adjusts recommendations based on hotel-specific settings. These settings include property type, segment focus, occupancy sensitivity, ADR sensitivity, RevPAR priority, rooms-sold priority, revenue priority, demand factors, seasonality, event impact, and rate guardrails.

This gave the project a stronger connection to real revenue management decisions. Instead of only producing one generic recommendation, the system can now support different pricing strategies depending on the property, market conditions, and business goals.

I also improved the project structure by adding and updating files such as tailored.py, app.py, main.py, dataset_manager.py, test_tailored.py, and test_tailored_integration.py. These updates helped connect the tailored model to the existing pipeline while preserving the baseline outputs, KPI calculations, validation checks, YoY results, and rate recommendation workflow.

Demo and Deployment Work

Another major focus was making the project easier to demonstrate. I prepared the Streamlit app for a board-style review workflow with a clearer dataset panel, demo dataset loading, and a one-click pricing simulation path. This included work around sample files such as sample_data.csv, future_on_books_sample.csv, events_sample.csv, and budget_daily_sample.csv.

I also worked on deployment preparation for Streamlit Community Cloud. That required checking the repository setup, cleaning up project files, confirming configuration files, and making sure the app could be launched from a GitHub-hosted project. During this stage, I also selected the project brand name RateAnchor, which reflects the idea of anchoring hotel pricing recommendations to real performance data.

What Went Right

The biggest success this month was moving the project from a technical prototype toward a usable web application. The baseline model, KPI outputs, year-over-year comparison features, and tailored recommendation layer now work together as parts of one larger decision-support system.

Another positive outcome was improving the reviewer experience. A capstone project needs to be more than technically correct; it also needs to be understandable. The demo workflow helps show what the system does without requiring someone to inspect every Python file manually.

What Went Wrong

One challenge was making sure the demo data actually supported the full app workflow. Demo data has to be realistic enough to show the system working, but simple enough that it does not create confusing edge cases during a presentation.

Another challenge was the median rate logic. I noticed that using a single median rate for the full forecast was too broad because hotel pricing decisions are made day by day. This gave me a clear next improvement: make the median rate anchor more date-specific so each recommendation can better reflect the conditions for that stay date.

How I Can Improve Moving Forward

Moving forward, I want to continue refining the daily pricing logic so the tailored recommendation layer is more precise. The next improvement is to make the median rate calculation function on a day-by-day basis instead of applying one value across the entire forecast window.

I also want to keep improving the dashboard text, clean up the interface, and make sure the final version of the app clearly explains why a recommendation was made. This is important because a revenue tool should not only output a number; it should also help the user understand the business reason behind the recommendation.

What I Learned This Month

This month helped me better understand the difference between building a working calculation and building a usable software product. A model can produce numbers, but a useful decision-support system also needs clear inputs, reliable validation, understandable outputs, realistic sample data, and a smooth user interface.

I also learned how important deployment preparation is. Getting the code to run locally is only one part of the project. Preparing a Streamlit app for review requires thinking about repository structure, configuration, sample data, user flow, and how the project will appear to someone seeing it for the first time.

Readiness for the Next Phase

The project is now closer to a complete capstone-ready product. RateAnchor has a stronger recommendation engine, a clearer app workflow, working demo data goals, and a deployment path through GitHub and Streamlit. The next phase is to polish the interface, finalize the daily recommendation logic, and prepare the project for presentation and defense.
